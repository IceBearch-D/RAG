# 工业级 RAG（Retrieval-Augmented Generation）完整工作流程深度拆解

---

## 一、RAG 的核心动机与定位

### 1.1 为什么需要 RAG？

大语言模型（LLM）存在三个根本性缺陷：

| 缺陷 | 具体表现 | RAG 的应对 |
|---|---|---|
| **知识过时** | 训练数据有截止日期，无法获知最新信息 | 通过检索实时知识库注入最新信息 |
| **幻觉问题** | 模型会"编造"事实，尤其在专业领域 | 以外部证据为锚点，约束生成内容的事实基础 |
| **缺乏私域知识** | 不了解企业内部文档、个人数据等私有信息 | 对私有文档建库后进行检索增强 |

**一句话定位**：RAG = **外挂一个可动态更新的知识库**，让 LLM 在回答时"先查资料，再回答"，从而在不重新训练模型的前提下注入新知识。

### 1.2 RAG vs. Fine-tuning 的核心区别

```
Fine-tuning：把知识"蒸馏"进模型参数里 → 知识固化，更新成本高
RAG：       把知识"放在外部"，推理时检索注入 → 知识可热更新，成本低
```

面试关键点：两者不是互斥的。工业实践中常先用 Fine-tuning 调整模型的"风格和能力"（如中文理解、领域术语），再用 RAG 注入"具体事实知识"。

## 二、工业级 RAG 总体架构（三阶段）

RAG 的完整链路可拆解为 **三大阶段**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        离线阶段（Indexing）                       │
│  文档收集 → 文档解析 → 分块(Chunking) → 向量化(Embedding) → 入库  │
└─────────────────────────────────────────────────────────────────┘
                              ↓ 构建完成后的索引库
┌─────────────────────────────────────────────────────────────────┐
│                        在线阶段（Retrieval）                      │
│  用户Query → Query理解 → 检索(多路) → 重排序(Rerank) → 筛选Top-K  │
└─────────────────────────────────────────────────────────────────┘
                              ↓ 检索到的上下文片段
┌─────────────────────────────────────────────────────────────────┐
│                        生成阶段（Generation）                     │
│  Prompt组装(Context + Query) → LLM生成 → 后处理(引用标注/去重/校验) │
└─────────────────────────────────────────────────────────────────┘
```

## 三、离线阶段：文档索引构建（Indexing Pipeline）

这是 RAG 系统的**基础工程**，数据质量直接决定检索质量的上限。

### 3.1 文档收集与接入

```
数据源类型：
├── 结构化数据：数据库表、Excel、CSV
├── 半结构化数据：JSON、XML、Markdown、HTML
├── 非结构化数据：PDF、Word、PPT、扫描件、图片
└── 实时数据：API接口、网页爬虫、日志流
```

**工业级关注点**：
- **数据源接入**：需要统一的 Connector 抽象，支持增量同步（避免重复处理）
- **权限管控**：不同用户能检索到的文档不同，需要在索引层面做文档级/段落级的权限标记
- **版本管理**：文档更新时，需要能够定位并替换旧的索引条目

### 3.2 文档解析（Document Parsing）

这一步的目标：**将各种格式的文档统一转换为纯文本，并保留结构信息**。

```
PDF为例的解析流程：
┌──────────┐    ┌───────────┐    ┌───────────┐    ┌──────────┐
│ 版面分析  │ →  │ 文本提取   │ →  │ 表格识别   │ →  │ 图片OCR  │
│(Layout)  │    │(Text Ext.) │    │(Table Det.)│    │          │
└──────────┘    └───────────┘    └───────────┘    └──────────┘
     ↓                ↓                ↓                ↓
  区分标题/        提取文字        结构化还原         图中文字
  正文/页眉        内容            为HTML/MD         识别提取
```

**关键技术与工具**：
- **PDF 解析**：PyMuPDF、pdfplumber（文本型）；PaddleOCR、Tesseract + LayoutParser（扫描型）
- **版面分析**：LayoutLMv3、DocTr 等模型识别标题、段落、表格、图片区域
- **表格处理**：将表格转为 Markdown 格式或结构化 JSON，保留行列关系（这对后续检索至关重要）

**面试重点 —— 解析质量为什么重要？**

> 如果一个表格被解析成一行乱序的文字，那么分块后这段内容就丧失了结构信息，检索到后 LLM 也无法正确理解。工业级系统中，文档解析往往是**最耗人工优化的环节**。

### 3.3 文档分块（Chunking）—— RAG 中最核心的工程问题

#### 为什么需要分块？

Embedding 模型对输入长度有限制（通常 512~8192 tokens），而且**过长的文本会稀释语义**，导致向量表示不够精确。分块的本质是：**将长文档切分为语义完整、长度适中的片段**，使得每个片段能够被精确表示和检索。

#### 分块策略对比

| 策略 | 方法 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| **固定长度分块** | 按字符/token 数切分，如每512 tokens一块 | 简单、可控 | 可能切断句子或段落 | 快速原型 |
| **递归分块** | 按层级分隔符递归切分：`\n\n` → `\n` → `.` → 空格 | 保留自然段落结构 | 需要调参 | **工业级最常用** |
| **语义分块** | 计算相邻句子的语义相似度，在相似度骤降处切分 | 语义边界最精确 | 计算成本高 | 高质量要求场景 |
| **结构化分块** | 按文档结构切分：每个标题/章节/表格为一块 | 利用文档原有结构 | 依赖解析质量 | 技术文档、手册 |
| **滑动窗口分块** | 固定窗口 + 步长重叠 | 保证上下文连续性 | 索引量增大 | 法律/医学等上下文敏感场景 |

#### 递归分块的典型实现逻辑

```python
def recursive_split(text, chunk_size=512, overlap=64, separators=["\n\n", "\n", ".", " "]):
    """
    核心逻辑：
    1. 优先用大分隔符（段落）切分
    2. 如果某段仍然超长，则用下一级分隔符继续细分
    3. 相邻块之间保留 overlap 个字符的重叠
    """
    if len(text) <= chunk_size:
        return [text]
    
    sep = separators[0]
    splits = text.split(sep)
    
    chunks = []
    current = ""
    for split in splits:
        if len(current) + len(split) <= chunk_size:
            current += split + sep
        else:
            if current:
                chunks.append(current.strip())
            # 如果单个split仍超长，递归用下一级分隔符
            if len(split) > chunk_size and len(separators) > 1:
                sub_chunks = recursive_split(split, chunk_size, overlap, separators[1:])
                chunks.extend(sub_chunks)
            else:
                current = split + sep
    if current.strip():
        chunks.append(current.strip())
    
    # 添加重叠
    return add_overlap(chunks, overlap)
```

#### Chunking 的关键参数

```
chunk_size  → 每个块的最大 token 数（通常 256~1024）
chunk_overlap → 相邻块的重叠 token 数（通常 chunk_size 的 10%~20%）
```

**面试关键点 —— 如何选择 chunk_size？**

> - chunk_size **太大**：语义被稀释，检索精度下降（一个块里混杂多个主题）
> - chunk_size **太小**：上下文不完整，LLM 缺乏足够信息来生成准确回答
> - 工业实践中，通常**针对不同类型的文档设置不同的分块策略**，而非一刀切

### 3.4 向量化（Embedding）

#### 原理

Embedding 模型将文本映射到高维向量空间，使得**语义相近的文本在向量空间中距离接近**。

```
"如何退款" → [0.12, -0.34, 0.78, ..., 0.45]  (768维)
"申请退货流程" → [0.11, -0.31, 0.80, ..., 0.43]  (距离近 ✓)
"今天天气不错" → [-0.56, 0.22, -0.11, ..., 0.67]  (距离远 ✗)
```

#### 工业级 Embedding 模型选型

| 模型 | 维度 | 特点 | 适用场景 |
|---|---|---|---|
| BGE (BAAI) | 768/1024 | 中文能力强，开源可控 | 中文场景首选 |
| GTE (阿里) | 768/1024 | 中英文均衡 | 中英混合场景 |
| text-embedding-3 (OpenAI) | 256~3072 | 可变维度，API调用 | 对延迟不敏感的场景 |
| Cohere Embed v3 | 1024 | 多语言，支持压缩 | 多语言场景 |
| Jina Embeddings v2 | 768 | 支持8K长文本 | 长文档场景 |

#### Embedding 的工程细节

```
1. 前缀(Prefix)处理：
   - 查询时加 "query: " 前缀
   - 文档时加 "passage: " 前缀
   - 这是 BGE/GTE 等模型的推荐做法，能显著提升检索精度

2. 批量化处理：
   - 使用 batch encoding 提高吞吐量
   - GPU 推理时注意 batch_size 与显存的平衡

3. 归一化：
   - 输出向量做 L2 归一化后，内积 ≈ 余弦相似度
   - 大多数向量数据库默认使用余弦相似度
```

### 3.5 向量存储（Vector Database）

#### 核心能力

向量数据库不仅存储向量，还需要支持：
1. **高效的近似最近邻搜索（ANN）**
2. **元数据过滤**（如按文档来源、时间、权限过滤）
3. **混合检索**（向量检索 + 关键词检索）

#### 主流选型

| 数据库 | 特点 | 适用场景 |
|---|---|---|
| Milvus/Zilliz | 分布式，支持十亿级向量，丰富索引 | 大规模生产环境 |
| Qdrant | Rust编写，高性能，支持payload过滤 | 中大规模，对性能敏感 |
| Weaviate | 内置向量化，GraphQL接口 | 快速原型 |
| ChromaDB | 轻量，Python原生 | 原型验证 |
| Elasticsearch + vector | 兼容传统搜索 | 已有ES基础设施的团队 |
| FAISS (Meta) | 纯向量索引库，非数据库 | 离线/研究场景 |

#### ANN 索引算法原理

```
HNSW（Hierarchical Navigable Small World）—— 最常用
┌─────────────────────────────────────────────┐
│  层级结构：                                   │
│  Layer 2:  ○───────○                         │
│  Layer 1:  ○───○───○───○───○                 │
│  Layer 0:  ○─○─○─○─○─○─○─○─○─○─○─○          │
│                                              │
│  搜索过程：                                   │
│  从最高层的入口点开始，在当前层找到最近节点       │
│  → 下降到下一层，继续贪心搜索                   │
│  → 直到最底层返回最终结果                       │
│                                              │
│  时间复杂度：O(log N)，空间换时间               │
└─────────────────────────────────────────────┘
```

#### 存储设计

每个向量条目通常存储：
```json
{
    "id": "chunk_001",
    "vector": [0.12, -0.34, ...],        // 向量
    "metadata": {
        "doc_id": "doc_20240101",          // 原始文档ID
        "source": "product_manual.pdf",    // 来源
        "page": 15,                        // 页码
        "chunk_index": 3,                  // 块序号
        "created_at": "2024-01-15",        // 时间戳
        "permission": ["dept_engineering"]  // 权限标签
    },
    "text": "该产品的额定电压为220V..."      // 原始文本（用于展示和重排序）
}
```

## 四、在线阶段：检索（Retrieval Pipeline）

### 4.1 Query 理解与预处理

用户输入的原始 Query 往往不能直接用于检索，需要进行预处理：

```
用户原始输入: "那个东西怎么退"
         ↓
┌─ 1. Query改写(Rewrite) ─────────────────────────┐
│   "那个东西怎么退货退款"                            │
│   目的：补全口语化省略的关键词                       │
│   方法：LLM改写 / 同义词扩展                       │
└─────────────────────────────────────────────────┘
         ↓
┌─ 2. Query拆解(Decomposition) ───────────────────┐
│   复杂Query拆为多个子Query                         │
│   "对比A产品和B产品的续航和价格"                     │
│   → Q1: "A产品续航和价格"                          │
│   → Q2: "B产品续航和价格"                          │
│   目的：提高召回率，不同子问题可能命中不同文档         │
└─────────────────────────────────────────────────┘
         ↓
┌─ 3. 意图识别 ────────────────────────────────────┐
│   判断用户Query的类型：                             │
│   - 知识问答 → 走RAG                              │
│   - 闲聊 → 直接LLM生成                            │
│   - 操作指令 → 走工具调用                          │
│   目的：路由到正确的处理管道                         │
└─────────────────────────────────────────────────┘
```

#### HyDE（Hypothetical Document Embeddings）技术

这是工业级 RAG 中很常用的 Query 改进技术：

```
原理：
1. 让LLM先根据Query生成一个"假设性答案"（不需要正确，只是风格参考）
2. 用这个假设性答案去做Embedding，而非原始Query
3. 因为假设性答案的表述风格更接近文档内容，检索效果更好

示例：
原始Query: "产品保修政策是什么？"
HyDE生成: "本产品提供一年免费保修服务，在保修期内..."
用HyDE文本检索 → 命中率显著高于直接用Query检索
```

### 4.2 多路检索（Multi-Route Retrieval）

工业级 RAG 通常不会只用一种检索方式，而是**多路召回、融合排序**。

#### 路径一：向量检索（Dense Retrieval）

```
Query Embedding → 在向量数据库中做 ANN 搜索 → 返回 Top-K 最相似的文档块

相似度计算：
- 余弦相似度：cos(q, d) = (q · d) / (|q| × |d|)
- 内积（归一化后等价于余弦）：sim(q, d) = q · d
- L2距离：||q - d||²（越小越相似）

优点：语义匹配能力强，"退货"能匹配到"退款流程"
缺点：对精确关键词匹配较弱
```

#### 路径二：关键词检索（Sparse Retrieval / BM25）

```
BM25 核心公式：
                    IDF(qi) · f(qi, D) · (k1 + 1)
Score(Q, D) = Σ  ─────────────────────────────────
                 f(qi, D) + k1 · (1 - b + b · |D|/avgdl)

其中：
- f(qi, D)：词 qi 在文档 D 中的词频
- IDF(qi)：逆文档频率，衡量词的稀有程度
- |D|/avgdl：文档长度归一化
- k1, b：调节参数（通常 k1=1.2~2.0, b=0.75）

优点：精确关键词匹配能力强，"ERR-404"这类专有代码能精确命中
缺点：无法处理语义层面的同义替换
```

#### 路径三：知识图谱检索（可选，高阶）

```
Query → 实体识别 → 在知识图谱中图遍历 → 返回结构化关系

适用场景：需要推理关系链的问答
如"A公司的CEO毕业于哪所大学？"需要 A公司→CEO→毕业院校 的多跳推理
```

#### 为什么需要多路融合？

```
                    语义相似    精确匹配    推理能力
向量检索              ✅ 强       ❌ 弱       ❌ 无
BM25检索              ❌ 弱       ✅ 强       ❌ 无
知识图谱检索            ⚪ 中       ⚪ 中       ✅ 有

多路召回 = 取各路所长，最大化召回率
```

### 4.3 结果融合（Fusion）

多路检索返回的结果需要融合为统一排序：

#### RRF（Reciprocal Rank Fusion）—— 最常用

```
RRF公式：
RRF_score(d) = Σ 1 / (k + rank_i(d))

其中：
- rank_i(d)：文档 d 在第 i 路检索中的排名
- k：常数，通常取 60

示例：
文档A：向量检索排名 2，BM25排名 5
RRF(A) = 1/(60+2) + 1/(60+5) = 0.0161 + 0.0154 = 0.0315

文档B：向量检索排名 10，BM25排名 1
RRF(B) = 1/(60+10) + 1/(60+1) = 0.0143 + 0.0164 = 0.0307

→ A 排在 B 前面

特点：不需要归一化各路的分数，只依赖排名，简单有效
```

### 4.4 重排序（Reranking）—— 精排阶段

粗排（Embedding检索）用的是双塔模型（Query 和 Document 独立编码），表达能力有限。Reranking 使用**交叉编码器**对 Query 和候选文档联合编码，精度大幅提升。

```
粗排（Bi-Encoder）：              精排（Cross-Encoder）：
Query → [Encoder] → q_vec        Query + Doc → [Encoder] → 相关性分数
Doc   → [Encoder] → d_vec        (将Query和Doc拼接输入，捕获细粒度交互)
sim = cos(q_vec, d_vec)
                                    
优点：速度快（向量可预计算）       优点：精度高
缺点：精度相对低                   缺点：速度慢（每次都要联合编码）
```

#### 常用 Reranker 模型

```
- BGE Reranker (BAAI) —— 中文场景首选
- Cohere Rerank API —— 多语言
- Cross-Encoder (Sentence-Transformers) —— 英文场景
- bge-reranker-v2-m3 —— 支持中英多语言
```

#### Reranking 流程

```
向量检索 Top-50 → Reranker 精排 → 选出 Top-5

为什么不是直接检索 Top-5？
→ 因为粗排可能遗漏，扩大候选集再精排，能显著提高最终质量
```

### 4.5 后过滤与截断

```
Reranked结果
    ↓
┌─ 元数据过滤 ─────────────────┐
│ 过滤掉无权限的文档             │
│ 过滤掉过期版本的文档           │
└──────────────────────────────┘
    ↓
┌─ 相关性阈值过滤 ──────────────┐
│ Rerank 分数 < 0.3 的直接丢弃  │
│ 避免低质量片段污染上下文       │
└──────────────────────────────┘
    ↓
┌─ 去重与压缩 ──────────────────┐
│ 相似片段合并（MMR算法）        │
│ 保留多样性，避免信息冗余       │
└──────────────────────────────┘
    ↓
Top-K 最终片段（通常 K=3~8）
```

#### MMR（Maximal Marginal Relevance）多样性算法

```
MMR = λ × Relevance(q, d) - (1-λ) × max_sim(d, selected)

含义：在选择文档时，不仅考虑与Query的相关性，
还要惩罚与已选文档过于相似的候选者

效果：如果Top-1和Top-2内容几乎一样，MMR会选择Top-2替换为
一个相关但内容不同的文档，提高信息覆盖度
```

## 五、生成阶段（Generation Pipeline）

### 5.1 Prompt 组装

这是 RAG 生成质量的关键一步——如何将检索到的内容有效地注入 Prompt。

#### 标准 Prompt 模板

```
System Prompt:
"""
你是一个专业的客服助手。请严格根据以下参考资料回答用户问题。
如果参考资料中没有相关信息，请明确告知用户"根据现有资料无法回答"，不要编造。

回答要求：
1. 基于参考资料，准确回答
2. 引用来源（如：根据[文档A]第3页）
3. 语言简洁专业
"""

参考资料:
[1] 来源: product_manual.pdf, 第15页
内容: 本产品自购买之日起享受一年保修服务。保修范围包括...

[2] 来源: faq.html, 第3条
内容: 如需申请保修，请携带购买凭证前往最近的服务中心...

用户问题: {user_query}
```

#### Prompt 组装的关键设计原则

```
1. 参考资料编号标注：每个片段标记编号[1][2][3]，方便LLM引用
2. 来源标注：附上文档名和位置，增强可溯源性
3. 明确约束：告诉LLM"只能基于参考资料回答"，降低幻觉
4. 拒绝兜底：允许模型在信息不足时说"不知道"
5. 片段排列：通常按相关性从高到低排列
```

### 5.2 LLM 生成

```
输入: Assembled Prompt (System + Context + Query)
    ↓
LLM Inference (如 GPT-4, Qwen, DeepSeek, GLM等)
    ↓
输出: 基于上下文的生成回答
```

#### 工业级关注点

```
1. 模型选型：
   - 闭源API（GPT-4, Claude）：效果好，成本高，数据出境风险
   - 开源私部署（Qwen, DeepSeek, GLM）：可控，需要GPU资源
   - 工业实践中常混合使用：复杂问题用强模型，简单问题用轻量模型

2. Token 预算管理：
   System Prompt + 参考资料 + 用户问题 的总长度不能超过模型上下文窗口
   需要预留足够空间给生成部分
   经验值：Context 占 60%~70%，留给 Generation 30%~40%

3. 流式输出（Streaming）：
   工业场景通常用 SSE/WebSocket 流式返回，改善用户等待体验
```

### 5.3 后处理

```
LLM原始输出
    ↓
┌─ 引用验证 ─────────────────────────┐
│ 检查回答中的引用 [1][2] 是否真实存在于 │
│ 检索结果中，去除幻觉引用             │
└────────────────────────────────────┘
    ↓
┌─ 事实校验（可选） ─────────────────┐
│ 将生成的回答与检索片段做交叉验证      │
│ 检查是否有与原文矛盾的表述            │
└────────────────────────────────────┘
    ↓
┌─ 格式化输出 ──────────────────────┐
│ 添加参考来源列表                     │
│ 标注置信度                          │
│ Markdown格式化                      │
└────────────────────────────────────┘
    ↓
最终返回给用户的回答
```

## 六、工业级 RAG 进阶优化技术

### 6.1 Query 改写（Query Rewriting）深度方案

```
方法一：LLM改写
  原始Query → LLM("请将以下口语化问题改写为正式的技术查询语句") → 改写后Query

方法二：多Query生成（Multi-Query）
  原始Query → LLM("请生成3个不同角度的相关问题") → Q1, Q2, Q3
  → 分别检索，合并结果

方法三：Step-Back Prompting
  "这个零件怎么更换？" → "该产品的维护保养指南" (更高层级的问题)
  → 同时用原始Query和Step-Back Query检索
```

### 6.2 上下文压缩（Context Compression）

```
问题：检索到的片段可能包含大量与问题无关的信息，浪费Token

方案一：LLM压缩
  "请从以下段落中提取与问题'保修政策'相关的部分：{片段}"

方案二：LLMLingua（基于小模型的压缩）
  使用专门训练的小模型，将片段压缩为关键信息

方案三：LongLLMLingua
  对片段按与问题的相关度排序，高相关的保留原文，低相关的做摘要
```

### 6.3 Self-RAG（自适应检索）

```
不是所有问题都需要检索！

Self-RAG决策流程：
1. 判断是否需要检索 → 不需要则直接回答（如"1+1等于几"）
2. 如果检索，判断检索结果是否相关 → 不相关则换策略
3. 生成回答后，判断回答是否被检索内容支持
4. 判断回答是否有用

实现：用模型输出特殊token来标记每个决策
```

### 6.4 Agentic RAG（Agent驱动的RAG）

```
传统RAG：单轮检索 → 生成
Agentic RAG：多轮迭代，动态决策

┌─────────────────────────────────────────────┐
│                Agent Loop                    │
│                                              │
│  1. 接收用户问题                               │
│  2. 判断需要什么信息 → 制定检索计划              │
│  3. 执行检索 → 获得结果                        │
│  4. 评估结果质量                               │
│     - 信息不足 → 改写Query，重新检索            │
│     - 信息矛盾 → 扩大检索范围                   │
│     - 信息充足 → 进入生成                      │
│  5. 生成回答 → 验证                           │
│     - 回答不完整 → 回到步骤2                    │
│     - 回答满意 → 返回                          │
└─────────────────────────────────────────────┘

框架：LangGraph、LlamaIndex Workflows、AutoGen
```

### 6.5 图文混合RAG（Multimodal RAG）

```
处理包含图片、表格、公式的文档：

方案一：图片描述法
  图片 → VLM(视觉语言模型)生成文字描述 → 将描述作为文本块索引

方案二：多模态Embedding
  文本和图片统一编码到同一向量空间 → 联合检索
  如：CLIP、SigLIP

方案三：多模态LLM直接理解
  检索时保留原始图片 → 直接喂给GPT-4V/Qwen-VL等多模态模型
```

## 七、评估体系（Evaluation）

面试中经常会问："你怎么评估RAG系统的效果？"

### 7.1 分模块评估

```
┌─ 检索质量评估 ──────────────────────────────┐
│ Recall@K：Top-K结果中包含相关文档的比例        │
│ MRR (Mean Reciprocal Rank)：第一个相关结果的排名倒数  │
│ nDCG：考虑排名位置的评分                       │
│ 方法：准备标注数据集(问题-相关文档对)，自动计算   │
└──────────────────────────────────────────────┘

┌─ 生成质量评估 ──────────────────────────────┐
│ Faithfulness（忠实度）：回答是否忠于检索内容     │
│ Answer Relevance（相关性）：回答是否切题        │
│ Context Relevance（上下文相关性）：检索内容是否相关 │
│ 方法：RAGAS框架自动评估                        │
└──────────────────────────────────────────────┘

┌─ 端到端评估 ────────────────────────────────┐
│ 人工评估：准确性、完整性、可读性                │
│ LLM-as-Judge：用GPT-4等强模型自动打分          │
│ A/B测试：线上对比不同方案的用户满意度           │
└──────────────────────────────────────────────┘
```

### 7.2 RAGAS 框架（自动化评估标准）

```python
# RAGAS的核心指标
metrics = {
    "faithfulness":      # 回答中的每句话是否有检索内容支撑
    "answer_relevancy":  # 回答与问题的相关程度
    "context_precision": # 检索结果中相关内容的比例（精确率）
    "context_recall":    # 参考答案中的信息被检索结果覆盖的比例（召回率）
}
```

## 九、面试高频问题速查

| 问题 | 要点 |
|---|---|
| RAG的整体流程是什么？ | 离线索引（解析→分块→Embedding→入库）+ 在线检索（Query处理→多路召回→Rerank→筛选）+ 生成（Prompt组装→LLM→后处理） |
| 为什么需要分块？如何选择策略？ | Embedding长度限制+语义精度；递归分块最常用，不同文档类型用不同策略 |
| Embedding模型怎么选？ | 中文选BGE/GTE；关注维度、最大长度、中英文能力；需要在业务数据上做benchmark |
| 向量检索和BM25各自优缺点？ | 向量：语义匹配强，精确匹配弱；BM25：精确匹配强，语义匹配弱；工业级融合使用 |
| 什么是Reranking？为什么需要？ | Cross-encoder精排，比双塔粗排精度高；在粗排Top-50中选Top-5 |
| 如何评估RAG效果？ | RAGAS四指标（faithfulness/answer_relevancy/context_precision/context_recall）+ 人工评估 |
| RAG的幻觉问题怎么缓解？ | Prompt约束"只基于参考资料回答" + 引用验证 + 事实校验 + 允许拒绝回答 |
| 如何处理多跳推理问题？ | Agentic RAG多轮迭代 / 知识图谱 / Query分解后分别检索 |
| chunk_size 如何确定？ | 通常256-1024 tokens；在业务数据上做不同size的benchmark；不同文档类型可能需要不同参数 |
| RAG和微调怎么选？ | RAG适合注入事实知识、需要热更新；微调适合改变模型能力/风格；工业级常组合使用 |
